# supɃewhY - PTS

> 面向 GPT-5.6 的 Prompt 编译 Skill：把已有 Prompt 或粗略需求，整理成更符合 GPT-5.6 提示方式的“最小充分任务规范”。

`PTS` 是 **Prompt To Spec** 的缩写。

PTS 是围绕 GPT-5.6 这次模型更新制作的。OpenAI 的 GPT-5.6 指南强调：模型更能从上下文理解真实意图，提示词可以更精简，但仍应明确领域信息、硬约束、授权边界和成功标准。PTS 把这套方向做成可重复调用的 Skill。

它不是给 Prompt 堆术语、加角色或强行套模板，而是在不改变任务的前提下，重新整理目标、输入、边界、约束、交付物和完成标准。具体依据见 [OpenAI 的 GPT-5.6 模型指南](https://developers.openai.com/api/docs/guides/latest-model)。

> **Task immutable; prompt mutable.** 任务不变，Prompt 可以重写。

> **说明：** PTS 是由个人维护的公开项目，非 OpenAI 官方产品或官方 Skill。

## 为什么需要 PTS

很多 Prompt 的问题不在于“不够长”，而在于：

- 真正要完成的任务被客套话、角色包装和表演式思考淹没。
- 影响结果的输入、约束或完成标准没有说清楚。
- 为了显得专业，把一个简单请求扩张成不必要的研究、审计或多阶段流程。
- 改写过程中丢掉了目标平台、格式、构图、身份保持等硬约束。

PTS 的目标不是让 Prompt 看起来更专业，而是让它更准确、更可执行，同时保持原任务的规模和复杂度。

## 转换示例

### 示例一：删除表演式指令，保留真正的审查任务

转换前：

```text
你是一位拥有 20 年经验的世界顶级商业专家，请先深入思考，再模拟五位专家讨论，然后三轮反思，最后全面分析这份商业计划。需要指出关键风险、证据不足处和下一步建议。
```

PTS 后：

```text
请审查下面的商业计划，判断其核心逻辑和可行性。

【商业计划】
{{粘贴或上传商业计划}}

分析时，请区分计划中已有的事实、尚未验证的假设和缺失的证据，不要自行补造信息。

请输出：
1. 整体判断及主要依据。
2. 可能影响计划成立或落地的关键风险。
3. 证据不足、逻辑跳跃或仍需验证的地方。
4. 下一步最应该优先验证、补充或修改的事项。

如果现有材料不足以支持确定结论，请明确说明不确定性，以及哪些信息最可能改变判断。
```

这里删除的是不会改变可见结果的“专家身份、多人讨论、反思轮次”，保留并明确了真正需要的判断、风险、证据缺口和下一步建议。

### 示例二：图像编辑任务不丢失关键约束

转换前：

```text
把画面右边浅蓝色区域补充完整，不要有断层，保持左侧原有的画面不变。
```

PTS 后：

```text
请编辑提供的图片：仅补全画面右侧不完整的浅蓝色区域，使其与周围画面自然连续，不出现断层、接缝或明显色差。

保持左侧原有画面的内容、构图、颜色和细节完全不变。除补全右侧浅蓝色区域外，不修改其他部分。
```

这里没有为了“简洁”合并掉位置、颜色、连续性和左侧画面保护等结果关键约束。

## 安装

### 个人级安装到 Codex

```bash
git clone https://github.com/supbewhy-sudo/supbewhy-pts.git
cd supbewhy-pts

CODEX_SKILLS_DIR="${CODEX_HOME:-$HOME/.codex}/skills"
mkdir -p "$CODEX_SKILLS_DIR"
cp -R ./supbewhy-pts "$CODEX_SKILLS_DIR/"
```

### 项目级安装

在目标项目中创建 `.agents/skills/`，然后复制 Skill 运行时目录：

```bash
mkdir -p /path/to/project/.agents/skills
cp -R ./supbewhy-pts /path/to/project/.agents/skills/
```

也可以下载 [`dist/supbewhy-pts.zip`](dist/supbewhy-pts.zip)，解压后复制其中的 `supbewhy-pts/`。

安装后重启 Codex 或打开新任务，让客户端重新发现 Skill。

## 使用方式

### 最短调用方式

不想写完整指令时，把 `pts` 放在消息开头即可。英文大小写均可。

```text
pts:你是一位世界顶级专家，请深入分析这份商业计划……
```

```text
pts 把画面右侧的浅蓝色区域补充完整，不要有断层……
```

第一种用冒号直接连接待转换内容；第二种只在 `pts` 后加一个空格。`pts` 后必须有明确的 Prompt 或需求；单独输入 `pts`，或 `PTS` 明显表示其他业务缩写时，不触发转换。

### 直接使用 PTS 指令

```text
PTS 一下：你是一位世界顶级专家，请深入分析这份商业计划……
```

### 显式调用 Skill

```text
$supbewhy-pts 把下面这段提示词整理成最小充分任务规范：……
```

### 从一个粗略需求生成 Prompt

```text
把这个需求整理成一份可直接交给 GPT-5.6 的 Prompt：
我需要检查一份商业计划最容易被合作方质疑的地方。
```

### 要求解释修改原因

```text
PTS 一下下面的 Prompt，并说明原 Prompt 有什么问题、为什么这样修改：……
```

默认情况下，PTS 只返回一份可复制的最终 Prompt。只有用户明确要求时，才会追加问题分析和修改依据。

## PTS 会做什么

- 优化已有 Prompt。
- 根据明确需求从零生成 Prompt。
- 删除空泛角色、强度词、表演式思考和重复规则。
- 补齐执行所必需但确实缺失的输入、边界或完成标准。
- 保留任务的受众、用途、目标平台和结果关键约束。
- 在信息不足会改变结果时，把问题合并成一次提问。
- 用户禁止提问时，使用明确的 `{{占位符}}`，而不是擅自猜测。

## PTS 不会做什么

- 不以缩短 Prompt 为目标。
- 不把每个任务都套进固定模板。
- 不把快速检查升级成完整尽调、咨询项目或多阶段代理流程。
- 不补造事实、数据、用户偏好、权限或目标平台语法。
- 不因为作者标识 `supɃewhY`、`supBewhY` 或 `supbewhy` 单独出现就执行转换。
- 不把无关业务语境中的 `PTS`、`PST` 当成触发词。
- 不把“转换后直接执行任务”固化为 PTS 功能；PTS 在交付 Prompt 时结束。

## 核心原则

### 1. Scope Fidelity（范围忠实）

PTS 必须保持原任务隐含的深度、长度、交付数量、专业程度和使用场景。新增要求只有同时满足以下条件才会保留：

1. 解决真实缺口。
2. 明显提高结果的可用性、正确性或可验证性。
3. 确实适用于当前任务。
4. 删除后会实质降低结果质量。

### 2. 任务不变，表达可重构

PTS 可以彻底重写原文，但不能改变用户真正要完成的任务、目标受众和最终结果。

### 3. 结果关键约束不能压缩

目标平台、输出格式、证据要求、身份保持、构图、位置、颜色、比例和权限边界等约束，只要会影响结果，就必须分别保留。

### 4. 不制造虚假的确定性

缺少材料时提问；禁止提问时使用占位符；无法验证的事实不补造。

## 触发边界

应该触发：

- `pts:把下面这段提示词改得更适合 GPT-5.6……`
- `pts 把下面这个需求整理成 Prompt……`
- `PTS 一下这段提示词。`
- `优化这个 Prompt，让 GPT-5.6 更稳定地执行。`
- `根据下面的需求帮我生成一份 Prompt。`
- 在明确的提示词转换语境中，把 `PST` 视为 `PTS` 的可能误输入。

不应该仅凭关键词触发：

- `supɃewhY`、`supBewhY` 或 `supbewhy` 单独出现。
- `PTS`、`PST` 明显指时间戳、时区或其他业务缩写。
- 用户只要求执行任务，没有要求制作或修改 Prompt。
- 用户只要求普通翻译、摘要或改写，不涉及 Prompt 转换。

## 输出模式

| 场景 | 输出方式 |
| --- | --- |
| 默认转换 | 只输出最终 Prompt |
| 缺少会改变结果的信息 | 提出一次合并问题 |
| 用户要求解释已有 Prompt | 原 Prompt 问题、完整新 Prompt、关键修改依据 |
| 用户要求解释从零生成的 Prompt | 需求缺口与假设、完整 Prompt、关键构建依据 |
| 用户禁止提问 | 使用明确占位符和必要假设 |

## 验证

```bash
python3 scripts/validate_skill.py .
python3 -m unittest discover -s tests -v
```

当前自动检查覆盖：

- Skill 目录、YAML frontmatter（前置元数据）和 UI 元数据。
- `SKILL.md` 引用的参考文件是否真实存在。
- 可见发布目录和 ZIP 完整性。
- 15 个行为场景的数量、字段和风险类别覆盖。
- 隐藏 `.agents/` 发布根、缺失引用和错误案例数量等失败情况。

自动测试能够证明结构和发布契约一致，但不能代替真实模型行为评测。`tests/cases.json` 已建立 15 个评测场景；在它们被逐项运行前，本项目不会宣称“行为评测已通过”。人工评测方式见 [`tests/manual-eval.md`](tests/manual-eval.md)。
